# 2027 申请季后期申请管理系统 MVP 设计

## 1. 目标与边界

系统服务于留学后期申请团队，以 Excel 导入为主、系统编辑为辅。顾问继续使用本地 Excel；系统负责定期汇总、差异识别、筛选统计、风险提醒与月度反馈草稿。

MVP 采用本地单机运行方式，但数据访问层与用户模型需支持后续迁移至 PostgreSQL 多人版。系统不做复杂在线填报、客户自动发送、学校官网抓取、知识库或 OA 级权限。

## 2. 技术方案

- Web 框架：Next.js 全栈应用。
- 数据库：SQLite；通过 Prisma 隔离数据访问，预留 PostgreSQL 迁移能力。
- Excel：使用 `xlsx` 解析工作簿。
- 表格与筛选：TanStack Table。
- UI：Tailwind CSS 与轻量组件体系。
- 日期：date-fns。
- 测试：单元测试覆盖解析、匹配、差异和风险规则；集成测试覆盖导入事务；关键页面进行浏览器流程测试。

MVP 默认作为本地管理员使用，不实现复杂登录。数据模型仍保留 `User` 与 `Role`，修改日志中的操作者使用当前本地用户，为后续多人版保留升级路径。

## 3. 数据模型

### 3.1 Student

保存申请季、姓名、合同号、申请条线与类别、各阶段顾问、学校与课程背景、成绩摘要、特殊备注、客户风险标签、当前阶段、自动/人工/最终风险、归档状态及时间戳。

学生匹配优先级：

1. 系统学生 ID；
2. 合同号；
3. 标准化后的学生姓名 + 后期顾问 + 申请季。

同名、重复合同或多个候选匹配不得自动覆盖，必须进入待确认列表。

### 3.2 Application

每个学生的每所学校、专业、轮次形成一个申请单元，保存国家地区、学校中英文名、申请系统、学院、专业、轮次、DDL、要求、材料与文书、递交、Portal、结果、入读及风险跟进信息。

申请匹配优先级：

1. 系统申请单元 ID；
2. 学生 ID + 标准化学校名 + 标准化专业 + 标准化轮次。

无法唯一匹配时进入待确认，不自动合并。

### 3.3 ImportBatch 与 ImportIssue

`ImportBatch` 保存文件名、文件指纹、上传人、申请季、导入类型、Sheet 范围、统计、状态与时间。  
`ImportIssue` 保存字段缺失、日期异常、身份冲突、重复记录、疑似删除和敏感列忽略等问题及其处理状态。

导入批次状态包括：预览中、待确认、已提交、部分失败、失败、已取消。

### 3.4 ChangeLog

逐字段保存实体类型、实体 ID、字段名、旧值、新值、操作者、来源、导入批次和时间。密码及其他受限敏感值不得写入日志。

### 3.5 MonthlyReportDraft

保存学生、月份、生成选项、可编辑 Markdown 内容、状态、创建人、确认人和时间。报告仅生成草稿，不自动发送。

### 3.6 User 与 Role

MVP 内置本地管理员。模型预留管理员、流程组长、教研组长、加拿大组长、后期顾问和只读角色，后续多人版再启用登录与数据范围控制。

## 4. Excel 导入架构

导入采用两个阶段：

1. 解析与预览：读取工作簿、检测版式、匹配字段、标准化值、匹配现有实体并生成差异；
2. 确认与提交：用户处理冲突及疑似删除后，在数据库事务内一次性写入并生成日志。

取消预览不修改正式数据。正式提交发生关键错误时整体回滚，不保留半成品记录。

### 4.1 支持的版式

#### 申请季总表

一行一个学生，以表头名称而非列序号匹配。当前真实样表为单 Sheet、39 列，包含学生、顾问、合同、背景、交接和早期服务进度。

总表用于新增或更新学生主档。上传总表、顾问部分表或只包含部分学生的文件时，不根据未出现的学生推断删除。

#### 单学生申请 Sheet

通常一个学生一个 Sheet。当前模板包含 92 列：

- 1–38：学生基本信息与材料；
- 39–54：定校及申请要求；
- 55–78：递交、Portal 和结果；
- 79–92：确认入读步骤。

每所学校占一行，学生基本信息通常只在首行填写。解析时需要在同一 Sheet 内向下继承学生身份，不能因后续行学生姓名为空而丢弃学校记录。

未来允许一个工作簿包含多个学生 Sheet。每个 Sheet 独立识别、预览和报告问题。

### 4.2 表头匹配

匹配前执行：

- 去除首尾空格、换行与多余标点；
- 统一中英文全角/半角；
- 忽略大小写；
- 依据字段别名字典匹配；
- 同时参考分组表头与子表头，解决“学校名称”“申请轮次”等重复表头。

字段别名字典可在代码中维护并通过测试固定。未识别列保留在预览中，但不写入结构化字段。

### 4.3 值标准化

- Excel 日期序列、文本日期和真实日期统一转换；
- 学校、轮次、状态和结果进行别名归一；
- 原始值在导入预览中展示，正式字段保存标准值；
- 空单元格默认表示“未提供”，不覆盖数据库已有值；
- MVP 不提供批量“空值覆盖”开关，避免误清空；需要清空时在系统内手动编辑并记录日志。

### 4.4 敏感字段

包含“密码”“登录账密”等含义的列自动标记为敏感列并完全跳过。账号字段也不进入 MVP 数据库，避免与密码混合的单元格泄露。导入报告仅记录“已忽略敏感列及数量”，不保存或回显其内容。

月报、导出和日志均不得包含密码、合同金额、证件号等敏感信息。

### 4.5 新增、更新与疑似删除

- 数据库无、表格有：新增；
- 数据库有、表格有且非空字段变化：更新；
- 匹配不唯一：待确认；
- 表格状态明确为取消：标记取消；
- 数据库有、单学生完整 Sheet 中没有：疑似删除。

疑似删除只在以下条件全部满足时计算：

1. 明确识别出单个学生；
2. 明确识别出学校申请明细区域；
3. 用户确认该 Sheet 是该学生的完整申请范围。

用户可将疑似删除项处理为“本次未上传”“标记取消”或“确认删除”。默认不处理；正式删除采用软删除并保留日志。

## 5. 页面与交互

### 5.1 导航

固定左侧导航包含：管理看板、学生列表、申请单元、数据导入、导入记录、月度反馈、修改日志。

### 5.2 管理看板

展示学生数、申请单元数、已/未递交、录取数、红色风险、未来 7/14 天 DDL。包含顾问维度汇总、近期 DDL 风险和待处理导入事项。

### 5.3 数据导入

采用三步流程：

1. 选择申请季、导入类型、文件和 Sheet；
2. 查看字段匹配、原始预览、新增、更新、疑似删除、缺失字段、冲突和忽略的敏感列；
3. 处理必要确认后提交，并生成导入日志。

任何可能覆盖或取消已有记录的变更必须在第二步显式展示。

### 5.4 学生与申请

学生列表支持姓名、顾问、申请类别、阶段和风险筛选。学生详情按基础信息、背景与顾问、申请学校、风险记录、修改记录、月度反馈分组。

申请单元列表一行一所学校，支持顾问、学生、学校、日期范围、申请类别、状态、结果和风险组合筛选，并可导出当前筛选结果。

复杂背景与 92 列映射信息放在详情页或分组标签中，不在列表中横向铺满。

### 5.5 手动编辑

支持新增、编辑、归档学生；新增、编辑、取消和软删除申请单元。编辑使用表单或侧边抽屉，保存时生成逐字段修改日志。

### 5.6 月度反馈

用户选择学生、月份以及是否包含学校明细、风险和客户待配合事项。系统生成可编辑 Markdown 草稿，支持复制及导出 Markdown、Word 和 PDF。

## 6. 风险计算

保存 `systemRiskLevel`、`manualRiskLevel` 和 `finalRiskLevel`。人工等级存在时优先，否则采用系统等级。

导入或手动修改 DDL、材料、文书、递交、Portal、结果、入读和跟进字段后重新计算。规则按需求中的红、黄、绿条件实现，并把触发原因保存为结构化风险项。

所有涉及“长期未更新”的规则使用可配置阈值；MVP 默认文书 10 天、最近跟进 14 天。缺少补件截止日期或 I-20 长期阈值时，不强行判红，而是生成规则数据不足提示。

## 7. 月度反馈生成

系统根据指定月份内的 `ChangeLog` 提取已完成事项，根据当前未完成状态和未来 DDL 生成进行中事项、下月计划、待配合事项和风险提醒。

顾问填写的月度反馈备注优先进入“顾问建议”。无备注时生成规则化草稿，不调用外部 AI 服务。草稿可编辑、确认、复制和导出，但不自动发送。

## 8. 错误处理

- 无法识别表头：阻止提交并提示用户选择正确表头行；
- 学生或申请匹配冲突：进入待确认；
- 日期异常：保留原始值并提示，不覆盖现有日期；
- 重复行：在预览中合并提示或要求选择；
- 关键数据库错误：事务整体回滚；
- 非关键字段错误：跳过该字段并记录问题；
- 文件无法读取：导入批次标记失败，不创建学生或申请数据。

## 9. 测试与验收

从两份真实样表的结构制作脱敏测试夹具，不复制真实学生敏感信息。测试覆盖：

- 列换序、缺列、额外列和表头别名；
- 分组表头、重复表头与一个学生多所学校；
- 学生身份向下继承；
- 多 Sheet 工作簿；
- 新增、更新、空值不覆盖、疑似删除；
- 同名学生、重复合同号和申请匹配冲突；
- Excel 日期序列及多种文本日期；
- 密码和混合账密列跳过；
- 风险等级与人工覆盖；
- 月报内容来源与敏感字段排除；
- 事务回滚与修改日志完整性；
- 筛选、手动编辑及导出关键浏览器流程。

验收以用户原始需求中的 12 项标准为准，并增加两项安全验收：

1. 导入预览取消后数据库无变化；
2. 密码内容不出现在数据库、日志、页面响应或导出文件中。

## 10. 实施顺序

1. 项目骨架、数据模型和测试基础设施；
2. Excel 版式检测、字段映射和标准化；
3. 实体匹配、差异预览、确认提交和日志；
4. 学生/申请列表、详情与手动编辑；
5. 风险引擎与管理看板；
6. 月度反馈草稿与导出；
7. 端到端验收、样表回归和本地运行说明。
